## Problem Statement

Piyush is a GenAI developer who builds systems and workflows for organizations (internal tooling, automation, "any gaps AI can fill"). He needs a personal portfolio website with one non-negotiable property: he regularly develops projects and must be able to add them continuously — live projects, dead projects, experiments — each with attached assets (YouTube videos, LinkedIn posts, images) and free-form descriptions, without ever touching layout code. Conventional portfolios rot because updating them is a code change; his must be a feed.

## What Makes This Cool

The homepage is a living particle node-graph that functions like an AI pipeline: each project is a node, data particles stream along the connections, a scroll-driven camera flies the visitor through the system, and the whole thing wears a dark terminal aesthetic. A GenAI systems developer whose site *behaves like* a GenAI system — visitors experience the craft instead of reading claims about it. The graph is not decoration: edges are computed at build time from real project metadata (shared tags/stack), so the visual is a truthful map of the work.

Dead projects are first-class: `status: dead` entries get a distinct post-mortem treatment. That honesty is rare in portfolios and reads as senior judgment to org buyers.

## Premises

1. **The portfolio is a feed, not a monument.** Adding a project is a content edit (one folder with markdown + assets), never a layout-code change. Supports live/dead/paused projects, YouTube embeds, LinkedIn posts, images, free-form MDX descriptions.
2. **The audience is org decision-makers arriving from LinkedIn, mostly mobile.** Spectacle serves credibility and must never cost legibility or load speed.
3. **The WebGL hero degrades gracefully.** Low-end devices and no-WebGL browsers get a build-time static fallback; the full simulation is progressive enhancement.
4. **Deploy target is Vercel, static-first**, matching the existing PR-preview workflow.

### Approach B: Living System (CHOSEN)

## Recommended Approach

**Approach B — Living System**, scoped to the "proof-of-pipeline" v1 chosen in session decision D2 (ship the hero + content system + 2–3 real projects rather than waiting for a full 6-project catalog or settling for a splash page — completeness of the system over completeness of the content at launch): particle node-graph hero + content system wired end-to-end + 2–3 real projects (including at least one dead project with post-mortem treatment).

Architecture spine:
- **Stack:** Astro 5 + bun, deployed to Vercel (PR branch → preview → merge). TypeScript throughout.
- **Content system:** `src/content/projects/<slug>/index.mdx` with typed frontmatter. Required: `title`, `status: live|dead|paused`, `date`, `tags[]` (string list), `stack[]` (string list), `outcome` (one-sentence result; for dead projects, what killed it). Optional: `org_context` (one-line client/org setting — v2 semantic-layer field, never blocks a Tuesday update), `links { youtube?, linkedin?, live?, repo? }` (URLs; LinkedIn is a link + screenshot, never an API embed), `hero_asset` (path to the card/node thumbnail). Additional images/media live in the project's own `assets/` subfolder and are referenced from the MDX body with relative paths. Adding a project = adding one folder. Zod schema validation via Astro content collections so a malformed entry fails the build loudly, not silently.

  Example entry (`src/content/projects/invoice-triage/index.mdx`):
  ```yaml
  ---
  title: Invoice Triage Copilot
  status: live
  date: 2026-03-10
  tags: [automation, document-ai, internal-tools]
  stack: [python, claude-api, supabase]
  outcome: Cut invoice processing from 4 hours/day to 20 minutes for a 30-person finance team.
  org_context: Mid-size logistics company, finance ops
  links: { youtube: "https://youtu.be/...", live: "https://..." }
  hero_asset: ./assets/hero.png
  ---
  Free-form MDX body. ![pipeline diagram](./assets/pipeline.png)
  ```
- **Graph compiler:** build step reads the collection and emits `graph.json`. v1 edge rule (default, decided): nodes = projects plus one central hub node ("PIYUSH / SYSTEMS"); edges = shared `tags` + shared `stack` entries, weight = overlap count; every project node additionally connects to the hub, so no node is ever isolated and a 3-project launch graph still reads as a pipeline rather than three orphaned dots. Tags and stack entries are normalized to lowercase slugs at compile time, and the build warns on any tag used by exactly one project (a build-local check needing no cross-build state — it guards against `Python`/`python`/`py` vocabulary drift silently disconnecting the graph). The force layout runs ONCE at build time: settled `x/y/z` coordinates are baked into `graph.json` alongside the establishing-view camera pose. Projection contract: the poster is an orthographic projection from that stored camera pose; the SVG uses `preserveAspectRatio="xMidYMid meet"` (contain), and the sim camera reproduces the identical orthographic mapping at every viewport size — so the poster→sim crossfade is seamless, with no node teleporting. An ambient non-project particle field surrounds the graph so small N never looks sparse. This file drives both the hero and the static fallback.
- **Hero island (v1 scroll behavior, defined):** the hero is a fixed-position three.js canvas occupying the first ~2 viewport-heights of scroll range. Camera choreography in v1 is ONE move: scroll progress (read from native scroll position) dollies the camera from a wide establishing view into the node cluster; past the hero range, the canvas releases and normal document scroll continues into the project feed below. Lenis smooth scrolling is a desktop-only enhancement layered on native scroll; on mobile/touch, native scrolling only — no scroll hijacking. three.js points + d3-force-3d, one `client:visible` island, custom point sprites/shaders for the pipeline-flow feel; nodes clickable → case-study page. Timebox: particle polish capped at one session for v1.
- **Fallback and device matrix (concrete gates):** the build emits a static constellation poster from the same `graph.json`, served as `<img src="constellation.svg">` (an `img` element, not inline SVG, so it is a valid LCP candidate). It is ALWAYS the initial paint on every device, and the sim fades in over it after hydration on capable devices — layout continuity guaranteed by the baked coordinates above. Gate rules: (a) no WebGL context OR `prefers-reduced-motion` → static only, never hydrate the sim; (b) viewport < 768px → static poster with a "run the system" tap affordance that hydrates the sim at half particle budget on demand; (c) `navigator.deviceMemory` < 4 where the API exists (Chromium-only; treat as absent on Safari/Firefox) → same tap-to-run treatment; (d) otherwise → sim auto-hydrates after first paint. Particle budget: start at 6,000 points on desktop, 3,000 on tap-to-run mobile; tune downward against Success Criterion 2 — the number is a starting point, the criterion is the contract. **Accepted trade, stated plainly:** the majority-mobile audience sees the poster by default and the full simulation is opt-in via tap. This is deliberate — the poster IS the real graph (same data, same layout), it loads instantly, and the tap turns spectacle into an interaction rather than a load-time tax. Layout reservation: the gate check runs in an inline head script before first paint. Static-only paths (gate a) lay out a single viewport-height hero. Paths that can or will run the sim (gates b/c/d) reserve the ~2-viewport-height scroll range at initial layout, so hydration never shifts content below (CLS-safe). Mobile tap-to-run keeps the sim inside its reserved canvas with the camera dolly disabled — the sim animates in place, no expansion, no layout jump.
- **Status treatments:** `dead` → node rendered desaturated/red-shifted in the graph; case page opens with a "POST-MORTEM" banner and the `outcome` field states what killed it. `paused` → node dimmed with an "on ice" badge on its card; standard case template otherwise. `live` → full treatment.
- **Project feed:** below the hero, a reverse-chronological (by `date`) list of ALL projects — dead and paused interleaved with live, distinguished by status badges, because the feed is the honest stream. Each card shows `title`, status badge, `outcome` line, `hero_asset` thumbnail, and tag chips, linking to the case-study page. Cards without `hero_asset` render a generated placeholder tile in the status color; graph nodes never depend on `hero_asset`. Every project is reachable from the feed with zero dependence on the canvas — the graph is a second way in, never the only way in.
- **Contact/CTA:** persistent site footer with `mailto:`, LinkedIn profile link, and a one-line "Work with me" CTA; case-study pages end with the same CTA block. A visitor can reach a contact action within one interaction from any page.
- **Case-study template:** dark, monospace-accented, mobile-first. YouTube via lite-embed facade (no iframe until click). LinkedIn posts as screenshot + link (the LinkedIn embed/API path is a tarpit — explicitly out of scope).
- **Design tokens:** background #0A0A0F, accent cyan #22D3EE, secondary violet #8B5CF6, white/gray monospace-accented type — matching the Flow concept video so the site and its marketing asset are visually continuous.

## Success Criteria

1. **The 15-minute Tuesday test (primary):** adding project #4 — folder, frontmatter, assets, push, live — takes under 15 minutes and touches nothing outside `src/content/projects/`. Only required fields need filling; optional fields never block an update.
2. Lighthouse mobile (throttled): homepage LCP under 2.5s on BOTH paths — the poster `<img>` is the LCP element everywhere, and on the WebGL path the sim hydrates after first paint without a long-task block > 200ms. Verification harness: Chrome DevTools, 4x CPU throttle, Fast 3G network profile.
3. A dead project renders with the defined post-mortem treatment (desaturated node, POST-MORTEM banner, `outcome` states what killed it).
4. The hero graph is generated from `graph.json`, not hand-placed — verified by adding a project and seeing the graph change without code edits.
5. A shareable production URL exists at the end of the proof-of-pipeline build with 2–3 real projects live.
6. `prefers-reduced-motion` visitors get the static constellation and no simulation — verified with emulation.
7. A contact action (mailto or LinkedIn) is reachable within one interaction from every page.
8. The mobile tap-to-run path works: on a 4x-CPU-throttled Chrome mobile profile, tapping "run the system" hydrates the half-budget sim without freezing scroll, and nodes appear exactly where the poster drew them (no layout jump).
9. Every project is reachable from the feed without the canvas — verified with JavaScript disabled.

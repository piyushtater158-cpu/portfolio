## Problem Statement

The approved Living System portfolio needs a way to add and edit projects continuously from anywhere — including a phone — without touching layout code. The builder proposed an admin panel on a separate domain. Session premises established: the portfolio ships first, projects remain files in the repo, and any admin surface writes git commits, never database rows. Given four evaluated options (GitHub-native workflow, git-backed CMS panel, agent-drafting workflow, or CMS+agent combined), the builder chose the zero-build option: **GitHub itself is the admin panel.**

## What Makes This Cool

The admin panel already exists, is maintained by GitHub, works on every device, and costs nothing. Adding a project from a phone is: open GitHub mobile → copy the template folder → fill in frontmatter → commit to a branch → tap the Vercel preview link → merge. Every "save" is a git commit with history, every edit gets a preview deployment before it goes live, and there is zero additional infrastructure that can rot, break, or need securing. This is subtraction as a feature: the builder rejected two recommended tooling layers because the leanest path preserves all upgrade options while adding no maintenance burden.

## Constraints

- Solo builder; updates must fit a 15-minute slot from any device (the approved doc's primary success metric).
- No new deployments, no OAuth shims, no databases — explicitly chosen over the CMS option.
- The portfolio repo (to be created outside OneDrive per the approved doc) is the single source of truth.
- Never push to main directly (existing workflow rule): all content edits go through a branch + Vercel preview.
- **The portfolio repo is public.** Branch protection with required status checks (Success Criterion 5) is free only on public repos — on a private repo it needs GitHub Pro. Public also suits a portfolio (the code is itself a work sample). If the builder later wants it private, GitHub Pro or dropping criterion 5's enforcement is a conscious trade, not a silent gap.
- Upgrade paths to a git-backed CMS (Sveltia/Pages CMS on an admin subdomain) and an agent-drafting workflow must remain open with zero rework.

## Premises

1. **Sequencing:** Portfolio v1 (proof-of-pipeline) ships first. Any *built* admin surface (a CMS panel or hosted agent endpoint) is v1.5 — designed now, built after. Approach A's artifacts (the `_template/` folder and README workflow section) are not a built surface; they are v1 scaffolding and ship with v1. (Agreed D1, clarified during review.)
2. **Architecture contract holds:** projects stay files in the repo; updates are git commits that trigger Vercel rebuilds. (Agreed D1.)
3. **The update surface solves friction, not content.** Case studies still get written by the builder. (Agreed D1.)
4. **A separate admin domain remains available later** but is not built now — the chosen approach needs no domain at all.

## Approaches Considered

### Approach A: GitHub-as-Panel (CHOSEN)
GitHub mobile app + github.dev editor on `src/content/projects/`, a committed `_template/` folder to copy, Vercel preview per branch. Effort S (zero build), risk low. Cons accepted knowingly: raw YAML on a phone keyboard; validation happens at build time (Zod schema fails the build loudly), not edit time.

### Approach B: Git-backed CMS on admin subdomain
Sveltia CMS or Pages CMS as a second Vercel project, GitHub backend, schema-mirroring config (status dropdown, tag chips, image uploads, pre-commit validation), optional PR-based editorial mode. Effort S-M (CC ~1 session), risk low-med (OAuth shim). Rejected for now: one more deployment to maintain for an audience of one; revisit trigger defined below.

### Approach C: Agent-drafting workflow
A `/add-project` Claude Code workflow: paste YouTube/LinkedIn links + rough notes → agent drafts the case-study MDX with valid frontmatter → opens a PR → builder approves the Vercel preview. Effort S (CC ~half session), risk low. Rejected for now as tooling-before-content; noted as the natural first upgrade because it attacks the writing bottleneck, which no form UI addresses.

### Approach D: B + C combined (was the session recommendation)
Overridden by the builder in favor of A — subtraction over tooling breadth. Recorded so future sessions know D was considered, not missed.

## Recommended Approach

**Approach A — GitHub-as-Panel**, implemented as part of portfolio v1 (no separate build phase):

- **Template folder:** the repo ships with `src/content/projects/_template/index.mdx` containing every frontmatter field with inline comments and one example asset reference. `_template` is excluded from the content collection glob so it never renders.
- **Phone workflow (documented in the repo README after live verification):** target flow — create `src/content/projects/<slug>/index.mdx` on a new branch with the template's contents, upload images to the same folder, open a PR, check the Vercel preview URL from the bot comment, merge. GitHub exposes three different surfaces with different capabilities (native mobile app: browsing, edits, PR review/merge; mobile-web github.com: new-file creation with the type-a-path folder trick and binary upload; `github.dev`: full editor, effectively desktop-only). There is no folder-copy operation on any surface — "copy the template" means copying one file's contents. The exact per-step surface assignment is determined by the live phone test (Next Steps 2) BEFORE the README section is written, so the README documents steps that are verified to work, not assumed.
- **Editing an existing project** is a strict subset of the add flow: open the project's `index.mdx` on any surface → edit → commit to a branch → PR → preview → merge. The native mobile app handles this end-to-end today; edits are the easy case.
- **Validation net:** the approved doc's Zod schema is the guardrail — a malformed entry fails the Vercel build on the PR (never in production) with a readable error, and the PR simply doesn't get merged until green. Build-time validation on a preview branch is edit-time validation, one step later and production-safe.
- **Upgrade triggers (decided now so the revisit is data-driven, not vibes):** adopt Approach C when the builder has skipped adding a finished project for more than two weeks because writing the case study felt heavy; adopt Approach B if more than two content-edit PRs in a month fail the schema check on frontmatter typos, or if a non-technical collaborator ever needs edit access.

## Open Questions

1. **Viability, not polish:** verify from the builder's actual phone that file creation, folder creation (path-typing trick), and image upload are possible at all, and on which surface (native app vs mobile web). The native GitHub app has historically not supported new-file creation or binary upload — if the whole flow falls back to mobile-web-in-desktop-mode, the 15-minute bar is at risk. Outright failure is immediate evidence for Approach B, or for a phone-variant of Approach C (message photos + notes to an agent that opens the PR). This verification happens BEFORE the README workflow section is written.
2. Domain for the future admin subdomain — inherits the approved doc's open question on the primary domain.

## Success Criteria

1. Adding a project end-to-end from a phone (new folder, frontmatter, one image, PR, preview check, merge) takes under 15 minutes — same bar as the approved doc's Tuesday test, now explicitly device-independent. Editing an existing project from the phone (subset flow) works via the native app.
2. A deliberately malformed frontmatter entry on a PR branch fails the preview build with a readable error and never reaches production — verified with the branch protection from criterion 5 in place.
3. The repo README contains the documented, live-verified phone workflow, and the `_template/` folder exists and does not render on the site.
4. Zero new deployments, domains, or credentials exist after this design is implemented.
5. Branch protection is enabled on `main`: PRs required, Vercel preview build required to pass before merge, including for the repo owner. Without this, criteria 2's "never reaches production" is unenforced — a red PR is freely mergeable and mobile commits can land directly on main.

## Distribution Plan

No new artifact. The workflow rides the existing (planned) GitHub → Vercel pipeline from the approved design. The template folder and README section ship inside portfolio v1.

## Next Steps

2. During v1 content entry, run the live phone test (Open Question 1): verify file creation, folder creation, and image upload per surface, and time the full add flow against the 15-minute bar. THEN write the README workflow section from what actually worked.
3. Revisit Approaches B/C only when an upgrade trigger fires — the triggers are written above so the decision is mechanical. A failed phone test in step 2 counts as an immediate trigger.
